1//! # metw-accounts-center service
2//!
3//! This crate is the abstract orchestrating microservice for metw.cc. It
4//! manages clients and data repositories, and  provides a library interface to
5//! use with a presentation layer.
6//!
7//! Applications storing encrypted cryptographic secrets and serving users'
8//! public keys to other services can use this service.
9//!
10//! This microservice must not be extended with non-authentication features.
11//! Functionality of this microservice is limited only to username, email, and
12//! password authentication.
13//!
14//! Goals:
15//! - Authentication Service: A service can use this microservice as an
16//!   authentication authority.
17//! - End-to-End Encryption (E2EE): Users have a public key-private key pair
18//!   and a master key. Their master keys and private keys are stored in the
19//!   server, encrypted on the client side. The microservice publicly serves
20//!   users' public keys; services that use this microservice for
21//!   authentication can use those public keys to verify tokens signed by
22//!   users.
23//!
24//! Non-Goals:
25//! - Profiles, posts, shares, etc. The scope of this service is only
26//!   authentication.
27//!
28//! ## System Design
29//!
30//! This project strictly adheres to the [Separation of Concerns][SoC] (SoC)
31//! principle. The service abstraction levels are:
32//! - [`repo`]: Defines traits for application storage, database, and cache
33//!   access.
34//! - [`service`]: Provides an interface for manipulating application state.
35//!   Elaborates on low-level transactions defined in the `repo` layer to the
36//!   upper layers.
37//! - [`handlers`]: The final level of abstraction to manipulate the
38//!   application state. This layer is a transport-agnostic front end for
39//!   the microservice.
40//! - [`State`]: The application state contains the services constructed from
41//!   repositories and clients. Handlers require a state instance to construct.
42//!
43//! ### Supporting Modules
44//!
45//! [`client`] is used for external integrations (e.g., mail provider,
46//! CAPTCHA).
47//!
48//! To ensure type-safety, the [`dto`] module defines the interface to
49//! communicate with `handlers`. [`dto::repo`] and [`entity`] are used for
50//! internal communication between `service` and `repo` layers.
51//!
52//! [`util`] is used for miscellaneous utilities that do not fit into
53//! categories above. Check out documenatations of the `util` for details.
54//! Conceptually, the [`token`] and [`id`] modules are re-exported at the crate
55//! root as they are too common.
56//!
57//! [SoC]: https://en.wikipedia.org/wiki/Separation_of_concerns
58
